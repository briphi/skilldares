## 2026-05-23 — PRD gap-batch decisions (refining/extending the brief)

1. **Hint scope:** once per question (each MC question can independently use one hint)
2. **Speed pool sizing:** 40 Type A + 40 Type B = 80 speed-round questions
3. **Speed Type A factors:** price and ABV only (no pour-size — user override from default)
4. **Speed Type B criteria types:** (a) items in [dish]; (b) items that are GF; (c) items in [menu section]
5. **Pre-game instructions:** **none** beyond the random pre-game message (user override — "instructions are obvious"). The Start button + the message is the entire start screen.
6. **Mid-game refresh:** lose progress, restart from round 1 (no resume; simpler)
7. **End-screen high-score treatment:** show final score, show personal best, celebrate when current beats previous best
8. **New 8th message pool:** `new-high-score` with ~20 messages. Updated total: **370 messages** across 8 pools (was 350 across 7).
9. **Score increment behavior:** animated count-up with a bounce
10. **Question selection algorithm:** uniform random from pool; structural constraint that rounds 1-15 are MC and rounds 16-30 are speed (with ~50/50 split between Type A and Type B)
11. **Error handling defaults:**
    - `localStorage` unavailable (e.g., private browsing) → game plays normally, high-score features silently disabled
    - Question JSON fails to load → friendly error screen in the personality voice ("Something broke, try refreshing")

## 2026-05-23 — Mid-finalize override: automated tests moved IN-scope

Original decision (brief): automated tests deferred / out of scope for v1.

**Override:** Automated tests are now IN scope at unit + component level. E2E browser-automation tests remain out of scope.

- **What's in:** Unit tests for pure logic (scoring, picker state machine, streak transitions, content validation) + component-render tests for the question UIs, feedback flow, and end screen, via Testing Library + a DOM environment (e.g., JSDOM).
- **What's out:** End-to-end browser-automation tests (Playwright, Cypress).
- **Bar:** All load-bearing FRs in F1–F7 have at least one corresponding test. No formal coverage % target.
- **Why:** User flagged during PRD finalize that test correctness mattered enough to belong in v1. Bar deliberately constrained to keep build cost reasonable for a learning project.
- **PRD changes:** Added NFR9. Removed "Automated tests" line from §9 Out-of-Scope, replaced with "End-to-end browser-automation tests" entry. Added test-framework choice to §10 Open Items for Architecture.
