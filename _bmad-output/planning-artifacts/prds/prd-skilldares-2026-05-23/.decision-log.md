## 2026-05-23 — Mid-finalize override: automated tests moved IN-scope

Original decision (brief): automated tests deferred / out of scope for v1.

**Override:** Automated tests are now IN scope at unit + component level. E2E browser-automation tests remain out of scope.

- **What's in:** Unit tests for pure logic (scoring, picker state machine, streak transitions, content validation) + component-render tests for the question UIs, feedback flow, and end screen, via Testing Library + a DOM environment (e.g., JSDOM).

- **Bar:** All load-bearing FRs in F1–F7 have at least one corresponding test. No formal coverage % target.
- **Why:** User flagged during PRD finalize that test correctness mattered enough to belong in v1. Bar deliberately constrained to keep build cost reasonable for a learning project.
- **PRD changes:** Added NFR9. Removed "Automated tests" line from §9 Out-of-Scope, replaced with "End-to-end browser-automation tests" entry. Added test-framework choice to §10 Open Items for Architecture.

## 2026-05-23 — Post-finalize override: content quality bar raised

Original decision (brief + PRD finalize): **Content quality strategy = "good enough" for v1. Some LLM hallucinations accepted as the cost of the learning-project experiment. A future LLM-driven question-correctness review pass deferred and out of v1 scope.**

**Override:** **No hallucinations** for factual content. Every question's correct answer MUST be traceable to a specific line in the menu source. Verification happens at authoring time (during the dev-story workflow for each content story), not deferred.

- **Trigger:** During Story 1.4 planning (MC question authoring), user explicitly said "I would really like no hallucinations if possible, double check everything." This raises the bar from the original PRD's stance.
- **Scope of override:**
  - **Applies to (factual content):** Story 1.4 (MC questions), Story 2.1 (Speed Order questions), Story 2.2 (Speed Select questions). These all reference menu facts (ingredients, prices, GF status, beer ABVs, etc.).
  - **Does NOT apply to (inventive content):** Story 1.5 (message pools), Story 3.1 (new-high-score pool). These are creative personality copy with no facts to hallucinate; voice consistency is the quality bar there.
- **Verification approach (built into each affected story):**
  1. Build a structured menu fact ledger from `data/menu_*.md` before generation
  2. Generate questions only from ledger entries (no invented ingredients/prices/etc.)
  3. Dedicated verification pass cross-references every committed question against the menu source
  4. Questions that can't be 100% verified are dropped, not shipped
- **Trade-off:** Authoring takes longer, but training-correct content avoids the worse outcome of confidently-stated wrong facts being learned by new servers.
- **PRD changes:** §6 "Quality strategy" bullet rewritten. §9 "An LLM-driven question-correctness review pass" entry rewritten to reflect that authoring-time verification replaces the deferred review.

## 2026-05-24 — Post-finalize: FR9.1 added (MC option order randomized per game)

Original PRD: FR9 described the option *set* (1 correct, 1 funny-wrong, 2 distractors) but said nothing about the *displayed order*. The MC content was authored with `correctIndex: 0` for 245/246 questions and `funnyWrongIndex: 2` for 246/246 — a convention that made authoring quick but assumed the runtime would shuffle.

**Bug discovered in production (Epic 1 deploy, Story 1.17):** because nothing in the codebase shuffled the options at render time, the correct answer always appeared in the upper-left quadrant and the obviously-wrong option always in the lower-left. A player could win every game by always tapping upper-left without reading the question.

**Fix:** Added Fisher-Yates shuffle in `src/lib/gameSetup.ts` (`shuffleMCQuestion`), applied during `selectEpic1Game`. Shuffle is per-question per-game (a fresh rng on Play Again produces a different layout).

**PRD changes:**
- **FR9 amended** with a closing sentence: "The four options describe the option *set*; their *displayed order* SHALL be randomized per question per game (see FR9.1)."
- **New FR9.1** added: "The four answer options SHALL be presented in a uniform-random order per question per game so that the correct answer (and the funny-wrong option) do not occupy a predictable quadrant. The authored `correctIndex` and `funnyWrongIndex` in the content JSON define the *identity* of the options, not their displayed positions. A given question replayed across multiple games MAY (and typically WILL) show its options in different orders."

**Why explicit:** the rule was implicit in the architecture (any quiz UI should shuffle) but wasn't called out; this missing requirement caused a real production bug. Documenting it prevents a future re-architect from inadvertently removing the shuffle (e.g., for a "performance optimization" or as part of a Speed-type refactor).

**Test coverage:** 7 new tests in `src/lib/gameSetup.test.ts` — `shuffleMCQuestion` contract (6) + integration test that a pool of all-zero `correctIndex` yields a spread across positions 0-3 (1). Existing 190 tests unaffected.
