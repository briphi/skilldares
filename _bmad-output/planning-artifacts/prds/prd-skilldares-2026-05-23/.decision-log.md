- **PRD changes:** Added NFR9. Removed "Automated tests" line from §9 Out-of-Scope, replaced with "End-to-end browser-automation tests" entry. Added test-framework choice to §10 Open Items for Architecture.

## 2026-05-23 — Post-finalize override: content quality bar raised

Original decision (brief + PRD finalize): **Content quality strategy = "good enough" for v1. Some LLM hallucinations accepted as the cost of the learning-project experiment. A future LLM-driven question-correctness review pass deferred and out of v1 scope.**

**Override:** **No hallucinations** for factual content. Every question's correct answer MUST be traceable to a specific line in the menu source. Verification happens at authoring time (during the dev-story workflow for each content story), not deferred.

- **Trigger:** During Story 1.4 planning (MC question authoring), user explicitly said "I would really like no hallucinations if possible, double check everything." This raises the bar from the original PRD's stance.
- **Scope of override:**
  - **Applies to (factual content):** Story 1.4 (MC questions), Story 2.1 (Speed Order questions), Story 2.2 (Speed Select questions). These all reference menu facts (ingredients, prices, GF status, beer ABVs, etc.).
  - **Does NOT apply to (inventive content):** Story 1.5 (message pools), Story 3.1 (new-high-score pool). These are creative personality copy with no facts to hallucinate; voice consistency is the quality bar there.
- **Verification approach (built into each affected story):**
  1. Build a structured menu fact ledger from `data/menu_*.md` before generation
  2. Generate questions only from ledger entries (no invented ingredients/prices/etc.)
  3. Dedicated verification pass cross-references every committed question against the menu source
  4. Questions that can't be 100% verified are dropped, not shipped
- **Trade-off:** Authoring takes longer, but training-correct content avoids the worse outcome of confidently-stated wrong facts being learned by new servers.
- **PRD changes:** §6 "Quality strategy" bullet rewritten. §9 "An LLM-driven question-correctness review pass" entry rewritten to reflect that authoring-time verification replaces the deferred review.

## 2026-05-24 — Post-finalize: FR9.1 added (MC option order randomized per game)

Original PRD: FR9 described the option *set* (1 correct, 1 funny-wrong, 2 distractors) but said nothing about the *displayed order*. The MC content was authored with `correctIndex: 0` for 245/246 questions and `funnyWrongIndex: 2` for 246/246 — a convention that made authoring quick but assumed the runtime would shuffle.

**Bug discovered in production (Epic 1 deploy, Story 1.17):** because nothing in the codebase shuffled the options at render time, the correct answer always appeared in the upper-left quadrant and the obviously-wrong option always in the lower-left. A player could win every game by always tapping upper-left without reading the question.

**Fix:** Added Fisher-Yates shuffle in `src/lib/gameSetup.ts` (`shuffleMCQuestion`), applied during `selectEpic1Game`. Shuffle is per-question per-game (a fresh rng on Play Again produces a different layout).

**PRD changes:**
- **FR9 amended** with a closing sentence: "The four options describe the option *set*; their *displayed order* SHALL be randomized per question per game (see FR9.1)."
- **New FR9.1** added: "The four answer options SHALL be presented in a uniform-random order per question per game so that the correct answer (and the funny-wrong option) do not occupy a predictable quadrant. The authored `correctIndex` and `funnyWrongIndex` in the content JSON define the *identity* of the options, not their displayed positions. A given question replayed across multiple games MAY (and typically WILL) show its options in different orders."

**Why explicit:** the rule was implicit in the architecture (any quiz UI should shuffle) but wasn't called out; this missing requirement caused a real production bug. Documenting it prevents a future re-architect from inadvertently removing the shuffle (e.g., for a "performance optimization" or as part of a Speed-type refactor).

**Test coverage:** 7 new tests in `src/lib/gameSetup.test.ts` — `shuffleMCQuestion` contract (6) + integration test that a pool of all-zero `correctIndex` yields a spread across positions 0-3 (1). Existing 190 tests unaffected.

**Trigger commit:** `0bf5f3a` "Fix MC bug: shuffle question options at game-setup time".

## 2026-05-24 — Post-finalize: FR11.1 added (MC reveal sequence before feedback overlay)

Original PRD: FR11 said "Selecting an option SHALL immediately lock the answer ... and trigger the feedback flow." The word "immediately" was ambiguous — it could mean "lock-on-tap with no confirmation" (intended) OR "instant transition to feedback overlay" (which is what shipped).

**Bug discovered during gameplay (post-Story 1.17 deploy):** Story 1.13 designed a full post-answer reveal state in QuestionMC (✓ on correct, ✗ on wrong-picked, muted on others). But Story 1.15's integration in GameScreen dispatches ANSWER_QUESTION synchronously on tap, which immediately flips `state.phase` to `'feedback'`, unmounts QuestionMC, and mounts FeedbackOverlay. The reveal state was essentially dead code — players never saw what the correct answer was if they got it wrong.

**Fix:** QuestionMC now drives a two-stage reveal sequence (lock → reveal) via internal timers before calling `onAnswer`. The dispatch (and therefore the GameScreen render flip) is delayed until the reveal completes.

**Design decisions made during the fix:**

1. **Two-stage reveal (lock → reveal) rather than one-stage** for a more deliberate Beat-2 feel. The tap registers as a brief solo highlight before the wrongs dim.
2. **All wrong options dim equally** post-reveal (no separate red/✗ treatment for the player's wrong pick). The teaching value is "the right answer was THIS" — the player's pick is secondary, and the feedback overlay's verdict serves that purpose. This is a deliberate departure from Story 1.13's original design (which had a `wrongSelected` state with red outline + ✗).
3. **Total reveal duration: 1500ms** (400ms lock + 1100ms reveal). Long enough for the player to register the correct answer, short enough to not feel slow.
4. **Auto-advance after reveal** (no user tap to transition to FeedbackOverlay). The feedback overlay then has its own 400ms-delayed Next button for the user-paced advance to the *next* round.

**PRD changes:**
- **FR11 amended** with a clause pointing to FR11.1: "The transition to the feedback flow SHALL pass through a brief reveal sequence (FR11.1) so the player sees the correct answer before the verdict overlay appears."
- **New FR11.1** added describing the two-stage reveal sequence in detail (lock phase ~400ms + reveal phase ~1100ms + auto-advance).

**Why explicit:** the original FR11 left a real UX gap (player can't see the right answer if they got it wrong). The new spec captures the intended behavior precisely so any future re-architect doesn't strip the timers as "unnecessary delay".

**Test coverage:** 2 new tests in `src/components/game/QuestionMC.test.tsx` covering the two-stage reveal with fake timers (`fireEvent.click` instead of `userEvent.click` to avoid the fake-timers/user-event-v14 deadlock — same pattern as Story 1.15's debug log). Existing QuestionMC tests now pass `revealDurationMs={0}` / `lockDurationMs={0}` for synchronous behavior. Total: 199 tests (was 197).

**Trigger commit:** see the next fix commit on `main`.

## 2026-05-24 — Post-finalize: FR11.1 reveal indicator refined (no ✓ overlay)

**Issue noticed in production:** the reveal phase placed a large centered ✓ overlay on the correct quadrant (display font, 3xl size, full inset). This made the answer text unreadable during the reveal — the player saw the ✓ but couldn't read what the correct answer actually was. Defeated the entire purpose of the reveal phase.

**Fix:** removed the ✓ overlay element from QuestionMC. The correct answer is now indicated by:
1. Bright color (vs 25% opacity on all wrongs — increased contrast)
2. Success-green outline (5px, slightly thicker than before)
3. Slight scale-up (1.04 vs default 1.0)
4. Subtle green glow (box-shadow)

The combination is unmistakable without covering any text.

**FR11.1 wording update:** changed "full color + ✓ overlay" → "outline + brightness contrast (full color, thicker success-color outline, slight scale-up, subtle glow)" with an explicit note that no center overlay badge is used because it would obscure the option text.

**Why explicit:** a future re-architect might re-add a center badge for "clarity" without realizing it covers text. The PRD now documents the reason it's NOT there.

**Test coverage unchanged:** existing QuestionMC tests query `data-quadrant-state` not the visual overlay, so no test updates needed. 199 tests still pass.

**Trigger commit:** see the next fix commit on `main`.

## 2026-05-24 — Post-finalize: FR11.1 asymmetric reveal duration (wrong gets longer)

**Issue noticed in production:** the reveal phase used a single 1500ms duration for both correct and wrong answers. For wrong answers, 1500ms isn't enough time to read the (newly-removed-badge, outline-only) highlighted correct answer before the feedback overlay takes over.

**Fix:** asymmetric reveal duration based on `isCorrect`:
- Correct: 1500ms (unchanged — minimal pause; player already knows they got it right)
- Wrong: 3000ms (~2x — gives ~2 seconds focused on the correct answer after the lock phase)

Implementation: QuestionMC now takes `correctRevealMs` (default 1500) + `wrongRevealMs` (default 3000) instead of a single `revealDurationMs`. GameScreen forwards both. Tests pass 0 for either to enable the sync path.

**PRD changes:**
- **FR11.1** rewritten to specify the asymmetric durations: correct = ~1500ms total, wrong = ~3000ms total. Lock phase (~400ms) is the same for both.
- Added explicit rationale: "The player already knows they got it right" vs "The player needs extra time to read the highlighted correct answer they missed."

**Test coverage:** 1 new test in QuestionMC.test.tsx specifically for the wrong-answer 3000ms timing (using fake timers + fireEvent.click). Existing correct-answer 1500ms test retained. Total: 200 tests (was 199).

**Trigger commit:** see the next fix commit on `main`.

## 2026-05-24 — Post-finalize: FeedbackOverlay polish (animate in, no button delay)

**Two issues noticed in production:**

1. **FeedbackOverlay appeared all at once** with no entry animation. Other screens (StartScreen) already use the `fadeIn` motion variant for their entrance; the overlay was inconsistent.
2. **NEXT button had a 400ms delayed reveal** (Story 1.14 design — UX spec line 261 said "Next/Finish button appears ≈400ms after verdict (read time without dead time)"). This caused two issues post-Story-1.13 fix:
   - The 400ms delay was redundant now that the MC reveal phase (1500ms correct / 3000ms wrong) provides ample read time BEFORE the overlay even appears
   - When the button finally mounted, it shifted the feedback message text upward — visible layout shift

**Fix:**
- Wrapped FeedbackOverlay's root in `motion.div` with the existing `fadeIn` variants from `src/lib/motionVariants.ts`. Overlay now fades in over ~250ms when it mounts.
- Removed the `useState`/`useEffect`/`setTimeout` button-reveal logic; removed the `revealDelayMs` prop entirely. Button is part of the initial mount — no layout shift, no delayed click affordance.

**Implementation:**
- `FeedbackOverlay.tsx` simplified — removed `useState`, `useEffect`, and `revealDelayMs` prop; wrapped root in `motion.div`.
- `FeedbackOverlay.test.tsx` — removed the 2 delayed-reveal tests + the `revealDelayMs={0}` prop from every test call. Added 1 test that the button is present immediately on mount. Net: -1 test (199 vs 200).

**Note on UX spec deviation:** UX spec lines 261, 404, 430 all mention the 400ms button reveal. That spec was written when the MC reveal phase didn't exist (Stories 1.14 design predates Stories 1.13/1.17 reveal-phase fix). The original purpose ("read time without dead time") is now served by the asymmetric MC reveal duration. UX spec is now a stale reference on this one detail — PRD FR11.1 is the authoritative timing source.

**Trigger commit:** see the next fix commit on `main`.
