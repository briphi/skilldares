# Decision Log — Skilldares Product Brief

Captures decisions, rationale, and overrides made during the brief workflow.

---

## 2026-05-23 — Initial discovery session

### Scope & audience
- **Target user:** new servers at Kildares pub during onboarding (first-week training)
- **Goal:** help them memorize the menu (food + drinks)
- **Continuity:** carries forward the concept from the 2026-05-22 brainstorming session, but the specific feature ideas from that session are NOT being carried — fresh feature set defined here

### Technology constraints
- **Language:** TypeScript
- **Architecture:** static web app, self-contained, no external API calls at runtime
- **Browser support:** modern Chrome + Safari, mobile + desktop responsive
- **Otherwise:** tech stack flexible
- **Deploy:** source in GitHub, deployed via AWS Amplify (push-to-deploy CI/CD with public HTTPS URL)

### Game loop
- **No login, no account** — start straight from the landing page
- **30 questions per game**, each a "round"
- **Rounds 1–15:** multiple choice questions
- **Rounds 16–30:** speed-based questions with 15-second timer
- **End screen:** total score + "Play Again" CTA
- **Play Again:** pulls a fresh random 30 from the pool; same questions can recur across plays (no seen-question avoidance)
- **Running score** displayed in upper-right throughout

### Question types

**Multiple choice (rounds 1–15)**
- 4 options: 1 correct, 1 obviously-wrong-and-funny (e.g., "rat feces" as a salad ingredient), 2 close-distractors (plausible)
- Question variety: ingredient listing, pricing, attribute/category (GF, etc.), creative shapes encouraged

**Speed Type A — drag-to-order (rounds 16–30, ~50% of speed rounds)**
- 3–5 squares of menu items, drag into correct order by some factor (price, ABV, etc.)
- Submit reveals the factor value below each square + correct/incorrect overall result
- All-or-nothing scoring

**Speed Type B — multi-select criteria (rounds 16–30, ~50% of speed rounds)**
- 5 squares of items; user selects all squares meeting a criteria (e.g., "items in the Street Corn Bowl")
- Correct set size: 1 → all 5; at least one always meets criteria
- Distractors are real menu items from adjacent dishes (close, not random)
- Submit reveals which selections were right/wrong
- All-or-nothing scoring (exact match = 5, anything off = 0)
- Timer expiry = wrong = 0

### Scoring
- **5 points** per correct answer
- **0 points** for wrong (no penalty)
- **Hint** (rounds 1–15 only): button greys out 1 random wrong answer; correct-after-hint = **2 points** instead of 5
- **No hint** in speed rounds

### Personality / feedback message system
- **Voice:** bold, irreverent, attitude. Can be harsh, profane, insulting. Can praise hard. Reference: "Fuck yea, you are amazing, and possibl[y] a genius."
- **Picker logic:** per-answer + streak-aware
- **7 message pools × 50 messages each = 350 total messages:**
  1. `right-no-streak` — correct answer, no active streak
  2. `wrong-no-streak` — wrong answer, no active streak
  3. `on-fire` — correct answer during 3+ correct streak
  4. `doing-bad` — wrong answer during 3+ incorrect streak
  5. `streak-broken` — wrong answer that ends a 3+ correct streak (the "you HAD it" moment)
  6. `comeback` — correct answer that ends a 3+ incorrect streak
  7. `pre-game-encouragement` — shown on the start/instruction screen ("Let's not be a dumbass, let's learn the menu")

### Content & authoring
- **Question pool:** ≥200 multiple choice questions + speed-round pool (sized to support 15 speed rounds with variety across plays — to be sized during PRD)
- **Source data:** `data/menu_front.md` and `data/menu_back.md`
- **Format (decided):** JSON files in `data/questions/` (`multiple-choice.json`, `speed-order.json`, `speed-select.json`); TypeScript types alongside in a `.ts` file, runtime validates against
- **Each question carries `menuRefs`** — list of menu item slugs/keys it draws from (enables traceability when menu changes)
- **Authoring source:** LLM-generated **once** via an external LLM chat session, output manually committed to the repo. No build-time generator script in v1 scope.
- **Quality strategy:** "good enough" for v1. Some LLM hallucinations accepted as the cost of the experiment. Future LLM-driven review pass deferred (out of v1 scope).

### UI / vibe
- **Animated and fun** — fade transitions on element in/out
- **Speed rounds urgency:** elements shake/vibrate as the 15-second timer runs low
- **Post-answer flow:** reveal correct/incorrect + feedback message → "Next" button appears (or "Finish" on round 30)
- **Visual identity:** fresh **game-energy aesthetic** is primary. Irish-pub colors / aesthetic as a bonus if they don't fight the energy. Disregard the existing `index.html` styling. Reference: `kildarespubwc.com` for actual Kildares brand palette.

### Persistence
- **localStorage for high score only.** Per-device, no account.
- **Implicit assumption:** each server plays on their own phone, not a shared device. If the device is shared, the high-score model breaks (degrades to "highest score anyone got on this device").

### Onboarding flow
- Landing/start screen → "Start Game" button + instruction text → round 1
- Start screen includes a random message from the `pre-game-encouragement` pool

### Explicitly out of scope (v1)
- Accessibility (keyboard nav, screen-reader support, color-contrast standards) — user opted out for v1
- Formal test suite — not specified; defaulting to no formal automated tests for v1
- Question-generator script (build-time tooling)
- LLM-driven question-correctness review pass
- Login, accounts, cross-device sync, leaderboards
- Per-question history, seen-question avoidance, adaptive difficulty
- Manager dashboard, admin tooling, content authoring UI

### Decisions reversing the 2026-05-22 brainstorming session
- The 8 "breakthrough concepts" from that session (SRS, allergen lifeguard, QR-on-menu, etc.) are NOT in v1 scope
- The "memory palace," "pub-map," and other novel mechanics from the brainstorm are NOT in v1 scope
- The concept (static quiz app for new servers learning the menu) is preserved; the specific mechanics are entirely fresh

- Exact content prompt design for the LLM author-time generation
- The shared-device assumption (validate by talking to actual Kildares manager about device usage on the floor)
- Specific UI framework choice (React? Svelte? Vanilla TS? Vite vs other bundler?) — not specified, deferred to PRD/architecture
- Where in the repo the build/deploy config lives (`amplify.yml`)
- The instruction screen content (what it actually teaches before round 1)
